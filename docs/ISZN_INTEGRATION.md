# Подключение «Гармонии» к серверу ИСЗН ЯНАО

Документ для команды, которая будет разворачивать backend и подключать приложение
к реальной базе ИСЗН на серверах ЯНАО. Описывает текущее состояние, что нужно
изменить и в каком порядке — без предположений о деталях, которых мы не знаем
(структура самой ИСЗН, протокол доступа и т.д. — это уточняется с владельцем системы).

---

## 1. Текущее состояние (важно понимать перед стартом)

Сейчас «Гармония» — **полностью статическое** веб-приложение:

- Все данные (услуги, сотрудники, новости, шаблоны писем) хранятся прямо в
  `js/data.js` как обычные JS-объекты.
- Правки через админ-панель сохраняются в `localStorage` **браузера конкретного
  устройства** — они не расходятся между пользователями и устройствами.
- Заявки/записи/такси отправляются через `mailto:` — открывается почтовый клиент
  **пользователя**, письмо не уходит с сервера автоматически.
- Авторизация администратора — проверка логина/пароля прямо в браузере
  (`js/admin.js`), пароль виден в исходном коде страницы.

Это осознанное решение для быстрого старта и демонстрации — но для реальной
эксплуатации с настоящими персональными данными граждан нужен backend.

## 2. Точка входа для интеграции — `REMOTE_API`

В начале `js/app.js` добавлена конфигурационная заготовка:

```js
const REMOTE_API={
  enabled:false,               // переключить в true, когда backend готов
  baseUrl:"",                  // напр. "https://iszn.yanao.ru/api/v1"
  endpoints:{
    branches:"/branches",
    services:"/branches/{city}/services",
    staff:"/branches/{city}/staff",
    orders:"/orders",
    bookings:"/bookings",
    taxiOrders:"/taxi/orders",
    auth:"/auth/login"
  }
};
```

Это **единственное место**, которое нужно менять на стороне фронтенда для
переключения с локальных данных на реальный сервер. Сама функция `apiRequest()`
уже написана и делает обычный `fetch()` с JSON.

## 3. Что должен отдавать backend (ожидаемый контракт)

Ниже — минимальный набор эндпоинтов, которые нужны фронтенду. Это **предложение**,
финальный контракт согласуется с владельцем ИСЗН.

### GET `/branches`
Список филиалов с контактами и режимом работы.
```json
[{"key":"gubkin","name":"Губкинский","address":"...","phone":"...","email":"...",
  "orderEmail":"...","openH":8,"openM":30,"closeH":18,"closeM":0}]
```

### GET `/branches/{city}/services`
Прейскурант филиала — категории и позиции. Структура **должна совпадать** с
текущей `GUBKIN_SERVICES`/`NOYABRSK_SERVICES` в `js/data.js`, чтобы не переписывать
весь рендеринг прейскуранта:
```json
[{"id":1,"name":"...","icon":"🏠","items":[{"n":"...","p":930,"m":884}]}]
```
`p` — базовая цена, `m` — цена по карте «Морошка» (`null`, если скидки нет).

### GET `/branches/{city}/staff`
Список сотрудников — структура как в `GUBKIN_STAFF`:
```json
[{"name":"...","pos":"Водитель","dept":"...","ext":"710","email":"..."}]
```
**Важно:** для модуля такси используются записи с `pos`, содержащим слово
«водитель» (без учёта регистра) — см. `getTaxiDrivers()` в `js/data.js`.

### POST `/orders`, `/bookings`, `/taxi/orders`
Приём заявки/записи/заказа такси. Тело запроса — то же, что сейчас уходит в
`mailto:`-письмо (см. функции `doSendOrder`, `taxiConfirmBooking` в `js/app.js`).
Backend должен сам отправить письмо на `orderEmail` филиала (SMTP), не полагаясь
на почтовый клиент пользователя.

### POST `/auth/login`
Логин администратора. Backend возвращает токен (JWT или подобное), фронтенд
хранит его в памяти на время сессии, прикладывает к последующим админ-запросам.
**Пароль администратора должен хешироваться на сервере** — сейчас в коде
(`js/admin.js`, объект `ADMIN`) он лежит открытым текстом, это нужно убрать при
переходе на реальный backend.

## 5. Размещение — требования по 152-ФЗ

Приложение обрабатывает персональные данные граждан (ФИО, телефон, СНИЛС) —
backend и база данных обязаны размещаться на территории РФ, с учётом требований
к защите персональных данных для государственных информационных систем.
Два реалистичных варианта:

- **Ведомственная инфраструктура ЯНАО** — сервер/ЦОД Правительства или
  Департамента, максимальный контроль, рекомендуется как основной вариант.
- **Аттестованное российское облако** (VK Cloud, Yandex Cloud, Cloud.ru, Selectel)
  с аттестацией по 152-ФЗ (УЗ-3 и выше) — как альтернатива, если собственная
  инфраструктура недоступна.

Фронтенд (статические файлы — `index.html`, `css/`, `js/`, `img/`) можно
размещать отдельно на любом CDN, поскольку сам по себе он не хранит персональных
данных — вся чувствительная информация уходит на backend только в момент запроса.

## 6. Порядок миграции (рекомендуемый)

1. Согласовать точный контракт эндпоинтов с владельцем ИСЗН (структура данных
   выше — стартовая точка для обсуждения, не финал).
2. Реализовать backend: приём заявок, авторизация, эндпоинты чтения данных.
3. Заполнить/подключить данные оставшихся 3 филиалов.
4. Включить `REMOTE_API.enabled=true`, указать `baseUrl`, протестировать на
   staging-окружении параллельно с текущей локальной версией.
5. Перенести админ-панель на серверную авторизацию (убрать пароль из кода).
6. Нагрузочное тестирование, обучение сотрудников центра, полный запуск.

## 7. Что НЕ нужно менять на фронтенде

Вся визуальная часть, помощник, дизайн, анимации — остаются как есть. Смена
источника данных со статического `js/data.js` на `REMOTE_API` не должна требовать
переписывания интерфейса, если backend отдаёт данные в описанной выше структуре.
